# Repository Instructions

## Portfolio Objective

This repository is part of Chuhan Shang's software-development portfolio. Recruiter-facing documentation should help present the project accurately and effectively when applying for software-engineering roles.

The primary audience is recruiters evaluating the author's experience, ownership, and engineering ability. Within one minute, a recruiter should be able to understand:

1. What the project does
2. What the author personally contributed
3. What technical ability and engineering judgment the project demonstrates

Optimize for clarity, credibility, and evidence rather than marketing. Present course projects honestly without diminishing their technical value or portraying assignment requirements as original product innovation.

## README Workflow

When creating or updating `README.md`:

* Inspect the repository before writing.
* Read `Docs/README_TEMPLATE.md` and use it as the default recruiter-facing structure.
* Tailor the structure to the project instead of filling every template section.
* Base all statements on verifiable repository evidence or explicit, reliable information supplied by the project owner.
* Remove irrelevant sections rather than leaving them empty or adding placeholder content.
* Verify all file paths, links, commands, schemes, targets, and configuration instructions.
* Identify and remove broken links, stale claims, and references to missing assets.
* Use professional English and keep the README concise enough to scan quickly.

## Recruiter-Focused Priorities

Prioritize content in this order:

1. Project title and a clear one-sentence summary
2. Screenshots, video, or another verified demonstration
3. Project context and the author's personal scope
4. Key implemented workflows
5. Technical highlights and engineering decisions
6. Architecture and data flow
7. Technology stack
8. Verified instructions for running the project
9. Author and professional profile links

Treat project status, known limitations, testing, deployment, API documentation, data models, roadmap, and license as optional sections. Include them only when they are relevant, supported by evidence, and useful to a recruiter.

## Ownership and Attribution

* State whether the project is individual, collaborative, course-based, capstone, or independent when that context matters.
* For individual work, claim independent design and implementation only when accurate.
* For team projects, describe the author's contribution without claiming work completed by others.
* For course projects, separate assignment requirements from independently designed features and engineering decisions.
* For tutorials, forks, starter projects, or derivative work, preserve attribution and explain the author's additions or changes.
* Never invent personal contributions, challenges, lessons, or project history.

## Content Standards

* List only implemented features and clearly separate them from planned work.
* Prefer five to seven representative workflows over an exhaustive feature inventory.
* Explain technical decisions in terms of the approach, reasoning, benefit, and relevant tradeoff.
* Do not use architecture labels such as MVVM or dependency injection unless the repository implements them.
* Do not invent features, commands, architecture, test coverage, metrics, APIs, benchmarks, or deployment links.
* Avoid generic claims such as "easy to use," "production-ready," "scalable," or "high performance" without evidence.
* Avoid AI-sounding filler, assignment-rubric narration, repeated information, and resume bullets duplicated in the README.
* Avoid unstable vanity metrics such as source-line or file counts unless they are meaningful, verified, and specifically requested.
* Use quantitative claims only when they are verifiable, useful to a recruiter, and reasonably durable.

## Demo Documentation

* Prefer three to five screenshots showing distinct, meaningful workflows or one short video or GIF showing the primary interaction flow.
* Use short captions that explain what each visual demonstrates.
* Reference only assets and links that exist and work.
* Omit the Demo section when no demonstration assets are available, but recognize missing visuals as a portfolio-readiness gap.
* Do not create, edit, or stage demonstration assets unless explicitly requested.

## Testing Documentation

* Include a Testing section only when executable automated tests exist in the repository.
* Do not describe previews, sample data, manual checks, or build success as automated tests.
* Include test commands only after verifying that they work for the repository.
* Mention test coverage only when a verifiable coverage report or configuration exists.
* If automated tests do not exist, omit the Testing section unless the absence of tests is a relevant known limitation.

## Security, Licensing, and Identity

* Never expose credentials, API keys, secrets, private data, or sensitive infrastructure details.
* Preserve accurate licensing and attribution information.
* Include a License section only when licensing information exists in the repository.
* Use the author name `Chuhan Shang`.
* Use the GitHub profile `https://github.com/shangc97`.
* Include Portfolio or LinkedIn links only when valid URLs have been supplied.

## Change Scope

Inspect the entire repository when necessary, but keep edits limited to recruiter-facing documentation directly required by the user's request.

Files that may be updated when relevant include:

* `README.md`
* `Docs/README_TEMPLATE.md`
* `AGENTS.md`
* existing README demonstration assets explicitly included in the request

When the request is only to create or update the project README, normally modify only `README.md`.

Do not modify application logic, tests, project configuration, dependency files, credentials, or Git history as part of portfolio-documentation work unless explicitly requested.
